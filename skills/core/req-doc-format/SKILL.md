# CRS 文档格式规范

本文档是 CRS 需求管理系统中所有文档格式的**唯一定义源**。创建、更新、审查需求文档时，必须遵循此处定义的结构。

## 1. 目录与命名规则

### 目录结构

```
.requirements/<type>/<ID>/
├── meta.yaml          # 需求元数据
├── raw.md             # 原始需求描述
├── spec.md            # 统一主文档
├── plan.md            # 实施计划
├── test-cases.md      # 测试用例
└── .claude-context.md # Claude 上下文指引（系统生成）
```

### type → 目录映射

| type | 目录名 |
|------|--------|
| feature | features |
| bug | bugs |
| question | questions |
| adjustment | adjustments |
| refactor | refactors |

### ID 格式

`{前缀}-{YYYYMMDD}-{三位序号}`

| type | 前缀 | 示例 |
|------|------|------|
| feature | FEAT | FEAT-20260525-001 |
| bug | BUG | BUG-20260525-001 |
| question | QUES | QUES-20260525-001 |
| adjustment | ADJ | ADJ-20260525-001 |
| refactor | REF | REF-20260525-001 |

## 2. meta.yaml 格式

```yaml
id: FEAT-20260525-001
type: feature
title: 需求标题（首行描述，限100字符）
description: 完整需求描述
created: "2026-05-25T10:00:00.000Z"
status: planning        # planning | in_progress | review | done | closed
priority: medium        # low | medium | high | critical
mode: semi_auto        # quick | semi_auto | full_auto | conservative
tags: []
# 以下字段在阶段 3（优先级评估）后填充
# priority_detail:
#   level: P1
#   score: 7.2
#   breakdown:
#     business_value: 8
#     urgency: 6
#     dependencies: 4
#     effort: 8
#     risk: 8
#   rationale: "评估理由"
#   estimated_effort: "2-3天"
#   roi: "高"
```

### 写入时机

- **阶段 1**：创建基础字段（id, type, title, description, created, status, priority, mode, tags）
- **阶段 3**：追加 priority_detail 字段

## 3. spec.md 格式

```markdown
# {title}

## 元数据

- **ID**: {id}
- **类型**: {type}
- **状态**: {status}
- **优先级**: {priority_level} ({score}分)
- **创建时间**: {YYYY-MM-DD}

## 需求分析

### 背景与目标

{描述业务背景和要达成的目标}

### 用户故事

- 作为{角色}，我想要{功能}，以便{价值}

### 验收标准

1. {具体可验证的验收条件}

### 开放问题

- {待确认的问题}

## 设计方案

### 系统架构

{整体架构设计，可用文字或 ASCII/Mermaid 图}

### 核心组件

- **组件名**：{职责描述}

### 数据流设计

{数据如何在组件间流动}

### 接口定义

{API 接口或模块接口签名}

### 技术选型

| 选型 | 理由 |
| ---- | ---- |
| {技术} | {选择理由} |

### 错误处理

| 场景 | 处理策略 |
| ---- | -------- |
| {异常情况} | {处理方式} |

## 测试策略

### 测试范围

| 模块 | 测试类型 | 优先级 |
| ---- | -------- | ------ |
| {模块名} | {单元/集成/E2E} | {高/中/低} |

### 测试类型

- 单元测试：{覆盖范围}
- 集成测试：{覆盖范围}
- E2E 测试：{覆盖范围}

## 实施计划

参见 plan.md

### 依赖关系

{任务间的依赖说明}

### 风险与应对

| 风险 | 影响 | 应对 |
| ---- | ---- | ---- |
| {风险描述} | {影响} | {应对方案} |

## 变更历史

| 日期 | 变更内容 | 影响分析 |
| ---- | -------- | -------- |
| {YYYY-MM-DD} | {变更描述} | {影响范围} |
```

### 写入时机

- **阶段 1**：创建骨架（含元数据、空章节标题、变更历史首行）
- **阶段 2**：填充「需求分析」和「设计方案」全部子章节
- **阶段 3**：更新「元数据」区（优先级、评分、状态）
- **阶段 5**：更新「实施计划」引用 → "参见 plan.md"

## 4. plan.md 格式

```markdown
# {title} — 实施计划

> 需求 ID: {id} | 创建: {YYYY-MM-DD}

## 任务分解

| # | 任务 | 状态 | 预估 | 依赖 |
| - | ---- | ---- | ---- | ---- |
| 1 | {具体任务} | pending | {工时} | {依赖的#} |
| 2 | {具体任务} | pending | {工时} | {依赖的#} |

任务状态：pending | in_progress | done | blocked

## 里程碑

- [ ] 阶段1：{里程碑描述}
- [ ] 阶段2：{里程碑描述}
- [ ] 阶段3：{里程碑描述}

## 依赖清单

| 依赖项 | 类型 | 说明 |
| ------ | ---- | ---- |
| {依赖名} | {外部/内部/模块} | {依赖说明} |

## 风险清单

| 风险 | 影响 | 概率 | 应对 |
| ---- | ---- | ---- | ---- |
| {风险描述} | {高/中/低} | {高/中/低} | {应对方案} |

## 变更记录

| 日期 | 内容 |
| ---- | ---- |
| {YYYY-MM-DD} | {变更描述} |
```

### 写入时机

- **阶段 1**：创建骨架（含表头空行、里程碑占位）
- **阶段 5**：填充「任务分解」「里程碑」「依赖清单」「风险清单」

## 5. test-cases.md 格式

```markdown
# {title} — 测试用例

> 需求 ID: {id} | 创建: {YYYY-MM-DD}

## 测试范围

| 模块 | 测试类型 | 优先级 |
| ---- | -------- | ------ |
| {模块名} | {单元/集成/E2E} | {高/中/低} |

## 测试用例

### 正向用例

| ID | 场景 | 前置条件 | 步骤 | 预期结果 | 优先级 |
| -- | ---- | -------- | ---- | -------- | ------ |
| TC-{ID}-01 | {场景描述} | {前置条件} | 1. {步骤}\n2. {步骤} | {预期结果} | {高/中/低} |

### 异常用例

| ID | 场景 | 前置条件 | 步骤 | 预期结果 | 优先级 |
| -- | ---- | -------- | ---- | -------- | ------ |
| TC-{ID}-02 | {异常场景} | {前置条件} | 1. {步骤}\n2. {步骤} | {错误提示/降级处理} | {高/中/低} |

### 边界用例

| ID | 场景 | 前置条件 | 步骤 | 预期结果 | 优先级 |
| -- | ---- | -------- | ---- | -------- | ------ |
| TC-{ID}-03 | {边界场景} | {边界条件} | 1. {步骤} | {边界行为} | {高/中/低} |

## 变更记录

| 日期 | 内容 |
| ---- | ---- |
| {YYYY-MM-DD} | {变更描述} |
```

### 写入时机

- **阶段 1**：创建骨架（含表头空行）
- **阶段 4**：填充「测试范围」和三个用例表（至少各 1 条）

## 6. 各阶段与文档的对应关系

| 阶段 | 操作 | 目标文件 | 目标章节 |
|------|------|----------|----------|
| 1/5 解析+初始化 | 创建骨架 | spec.md, plan.md, test-cases.md, meta.yaml | 全部（骨架） |
| 2/5 深度分析 | 填充分析结果 | spec.md | 「需求分析」「设计方案」 |
| 3/5 优先级+质量 | 更新评估结果 | spec.md, meta.yaml | spec.md「元数据」, meta.yaml「priority_detail」 |
| 4/5 测试策略 | 填充测试用例 | test-cases.md | 全部 |
| 5/5 实施计划 | 填充任务分解 | plan.md | 全部，spec.md「实施计划」引用 |

## 7. 占位符约定

骨架文件使用 `<!-- TODO: {说明} -->` 标记待填充区域。填充后**必须删除**对应的 TODO 注释。

## 8. 变更记录约定

所有文档末尾的变更记录表在每次更新时追加一行，不修改已有记录。
